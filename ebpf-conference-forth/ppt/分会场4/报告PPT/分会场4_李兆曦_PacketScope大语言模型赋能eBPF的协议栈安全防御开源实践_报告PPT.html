02 eBPF 概述 + XDP 入口

内核态可编程 — 无需修改内核源码，JIT 编译为原生指令

eBPF 是什么?

  • Linux 内核虚拟机，支持安全沙箱执行
  • Verifier 静态验证: 无死循环、无越界访问
  • JIT 编译 → 接近原生 C 代码性能
  • 动态挂载到 XDP / kprobe / tracepoint 等 hook 点
  • 内核态与用户态通过 BPF Maps 共享数据
  • CO-RE (Compile Once, Run Everywhere) 跨版本兼容

XDP 入口程序

[OK] packetscope.slide.17 :: 04 核心数据结构

04 核心数据结构

conn_key 五元组标识 + conn_info 连接元数据 — BPF Map 的 Key/Value

conn_key: 连接标识

C // 五元组唯一标识一个连接 struct conn_key { __u32 src_ip; // 源 IP (网络字节序) __u32 dst_ip; // 目标 IP __u16 src_port; // 源端口 __u16 dst_port; // 目标端口 __u8 protocol; // IPPROTO_TCP=6 __u8 pad[3]; // 对齐填充 };

conn_info: 连接元数据

C // 连接的完整运行时信息 struct conn_info { __u64 ts_start; // 连接建立时间 (ns) __u64 ts_last; // 最近活动时间 __u64 bytes_sent; // 发送字节数 __u64 bytes_recv; // 接收字节数 __u32 packets; // 总包数 __u32 pid; // 进程 PID char comm[16]; // 进程名 __u8 state; // TCP 状态 __u8 flags; // 标记位 };

设计考量

内存对齐: conn_key 使用 3 字节 pad 确保 4 字节对齐， 避免 BPF verifier 拒绝和哈希碰撞。

comm 字段: 16 字节进程名直接在内核态通过 bpf_get_current_comm() 获取，无需用户态查询 /proc。

06 过滤引擎: 规则 + BPF Maps

06 Guarder: Fast Path / Slow Path

XDP 内核态纳秒级过滤 + Go 用户态 LLM 秒级分析，BPF Maps 双向桥接

Fast Path (XDP Kernel - ~100ns/pkt)

Packet → NIC Driver → XDP Hook → check_enhanced_filter_rules() → 遍历 filter_map (ARRAY, 32 rules) → 五元组 + 协议特定匹配 (TCP flags / ICMP type+code / inner packet) → XDP_DROP (丢弃) 或 XDP_PASS (放行) 同时: update_conn_tracking() → conn_map (LRU_HASH, 65536) 同时: update_perf_stats() → __sync_fetch_and_add 原子计数

Slow Path (Go Userspace + LLM - seconds)

Go goroutine 轮询 BPF Maps (default 10s interval) → /api/connections 读取 conn_map → /api/ai/analyze 触发 LLM 分析 → 格式化流量上下文为结构化 Prompt → 调用 OpenAI / Anthropic / DeepSeek API → 解析返回的 filter rules → FilterManager.UpdateRule() 写入 filter_map → XDP 程序立即使用新规则 (无需重载)

BPF Maps: 快慢路径的桥梁

conn_map

快路径写入连接状态
↓
慢路径读取分析

filter_map

慢路径写入规则
↓
快路径执行过滤

perf_stats_map

快路径写入统计
↓
慢路径读取监控

06 Guarder: 核心代码实现

XDP 快路径过滤函数 + Go API 端点实现

check_enhanced_filter_rules() 核心逻辑

C static __always_inline int check_enhanced_filter_rules( struct xdp_md *ctx, struct iphdr *ip, __u16 src_port, __u16 dst_port) { for (__u32 i = 0; i < MAX_RULES; i++) { struct filter_rule *rule = bpf_map_lookup_elem( &filter_map, &i); if (!rule || !rule->active) continue; // 五元组匹配 if (rule->src_ip && rule->src_ip != ip->saddr) continue; if (rule->dst_port && rule->dst_port != dst_port) continue; // 协议特定匹配 if (ip->protocol == IPPROTO_TCP) { // TCP flags 匹配 (SYN/RST/FIN) if (rule->tcp_flags && !(tcp->flags & rule->tcp_flags)) continue; } else if (ip->protocol == IPPROTO_ICMP) { // ICMP type+code + inner packet if (rule->icmp_type && rule->icmp_type != icmp->type) continue; } // 命中: 更新统计, 执行 action __sync_fetch_and_add( &rule->hit_count, 1); return rule->action; } return XDP_PASS; // 无匹配放行 }

Go API 端点实现

Go // GET /api/connections - 读取 conn_map func (s *Server) handleConnections( w http.ResponseWriter, r *http.Request, ) { conns := make([]Connection, 0) iter := s.connMap.Iterate() var key ConnKey var val ConnInfo for iter.Next(&key, &val) { conns = append(conns, toConnection(key, val)) } json.NewEncoder(w).Encode(conns) } // POST /api/ai/analyze - LLM 分析 func (s *Server) handleAIAnalyze( w http.ResponseWriter, r *http.Request, ) { // 格式化流量上下文 ctx := s.buildTrafficContext() prompt := formatAnalysisPrompt(ctx) // 调用 LLM API resp, err := s.llm.Chat(r.Context(), prompt) if err != nil { http.Error(w, err.Error(), 500) return } // 解析并下发规则 rules := parseFilterRules(resp) for _, rule := range rules { s.filterMgr.UpdateRule(rule) // XDP 立即生效，无需重载 } json.NewEncoder(w).Encode(rules) }

07 技术选型: BCC vs cilium/ebpf

PacketScope 同时使用两个 eBPF 框架 — 各取所长

维度 BCC (Analyzer) cilium/ebpf (Guarder)
语言 Python + C Go + C
编译方式 运行时编译 (LLVM) 预编译 CO-RE (BTF)
部署依赖 需要 LLVM + 内核头文件 单一 Go 二进制
适用场景 快速原型 / 监控分析 生产环境 / 高性能路径
PacketScope 用途 Analyzer 连接监控 Guarder XDP 防火墙
核心优势 开发速度快，调试方便 跨内核版本，无运行时依赖

设计哲学: Analyzer 侧重快速迭代和灵活分析，BCC 的 Python 绑定是最佳选择； Guarder 需要零依赖部署和生产稳定性，cilium/ebpf 的 CO-RE 特性更合适。 两者通过 BPF Maps 和 gRPC 无缝协作。

07 Future Work

PacketScope 的演进方向: Agent 时代的网络安全

AI Agent 行为监控

PacketScope 可以监控 AI Agent 的网络行为 (API 调用模式、数据外泄检测、异常通信识别)，为 Agent Safety 提供网络层可观测性

Agent-native 运行

Agent-to-Agent 安全

监控多 Agent 系统之间的通信，检测 Agent 间的异常交互模式，防止 Agent 被劫持或滥用

QUIC / HTTP3 支持

深度解析 QUIC 协议，支持 HTTP/3 流量的安全分析与过滤

CO-RE 全面迁移

Analyzer 从 BCC 迁移至 cilium/ebpf CO-RE，实现单二进制零依赖部署

分布式部署

K8s DaemonSet 集群级部署，多节点威胁关联分析与统一策略管理
